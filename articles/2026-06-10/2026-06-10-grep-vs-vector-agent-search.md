# Grep 比向量检索更强？Agent 搜索的基准测试研究

**来源：** arXiv:2605.15184 "Is Grep All You Need? How Agent Harnesses Reshape Agentic Search"
**作者：** Sahil Sen, Akhil Kasturi, Elias Lumer, Anmol Gulati, Vamse Kumar Subbiah
**提交时间：** 2026-05-14
**HN 排名：** #13 (113 points, 52 comments)

---

## 核心问题

随着 LLM agent 能力越来越强，它们开始自主获取信息、调用工具、在大型代码库中搜索内容以完成任务。检索增强生成（RAG）成了 agent 搜索系统的标配。

但这里有个被忽略的问题：**在 agent 环境中，不同的检索策略（如 grep、向量检索）到底效果如何？agent 的架构和工具调用范式会不会扭曲这些比较结果？**

论文选择了更务实的研究路径：在真实 agent 环境中直接测试，而不是只跑离线检索基准。

---

## 研究方法

论文设计了两个实验：

### 实验 1：grep vs 向量检索

- **数据集：** LongMemEval 的 116 个问题
- **检索方式：** grep（关键词搜索） vs 向量检索（embedding 相似度）
- **工具输出方式：** 内联结果（inline） vs 文件式结果（file-based，模型单独读取）
- **Agent Harnesses：**
  - 自定义：Chronos
  - Provider 原生：Claude Code CLI、Codex CLI、Gemini CLI

### 实验 2：干扰环境下的性能

- 场景：在搜索查询中混入越来越多无关的对话历史
- 比较：纯 grep vs 纯向量检索
- 目标：看当周围有大量干扰文本时，检索策略的鲁棒性如何

---

## 主要发现

### 1. grep 通常比向量检索更准确

在 Chronos 和三个 provider CLI（Claude Code、Codex、Gemini）的测试中，**grep 整体上比向量检索获得了更高的准确率**。

这个结论有点反直觉——大家通常认为向量检索（RAG）更"先进"，grep 更"原始"。但结果却是 grep 赢了。

### 2. Harness 和工具调用样式影响巨大

即使底层对话数据相同，**agent harness 和工具调用样式的选择也会对最终得分产生显著影响**。

这意味着：
- 同一个 agent，换个 CLI 工具可能性能差异很大
- 工具输出如何呈现给模型（内联 vs 文件式），可能比检索策略本身更重要
- 纯粹比较"向量检索 vs grep"没有意义，必须放在具体 agent 环境中测试

### 3. 论文未详细说明实验 2 结果

摘要只描述了实验 2 的设计（逐渐混入无关对话历史），没有报告具体结果。因此，**grep 和向量检索在干扰环境下的鲁棒性对比，尚无法从摘要中得出结论**。

## 为什么 grep 可能更有效？

摘要没有深入解释原因，但可以推测：

1. **代码搜索特性：** LongMemEval 可能面向代码相关问题。代码中的关键词、标识符、函数名比自然语言语义更容易被 grep 精准命中。
2. **向量检索的噪声：** embedding 相似度可能匹配到语义相关但无关的段落，而 grep 严格匹配关键词。
3. **Agent Loop 中的信息密度：** agent 不需要阅读整个代码库，只需要找到精准的上下文片段。grep 的精确匹配可能更符合这个需求。

## 对 Agent 工程的启示

### 1. 不要迷信"先进"检索策略

- 向量检索是 RAG 的默认选择，但 **在代码搜索场景下，grep 可能更高效**。
- 需要针对具体任务类型（代码搜索、文档搜索、问答）设计检索策略，而不是一刀切用向量。

### 2. Harness 的选择很重要

- 同一个 agent，在不同 CLI 工具中可能有截然不同的表现。
- 如果你的 agent 在某个环境中表现差，**可能不是算法问题，而是工具调用方式的问题**（内联输出 vs 文件输出、上下文窗口管理、工具链设计）。

### 3. 需要更多 Agent 环境中的基准测试

- 论文的核心贡献是：**在真实 agent 环境中测试检索策略，而不是离线检索基准**。
- 这为 agent 工程提供了更贴近实际的测试框架。

## 未知与待验证

基于摘要信息，仍有以下关键问题无法回答：

1. **grep 的优势范围：** 这个结论是否只适用于代码搜索场景？如果是自然语言问答、知识检索，向量检索可能更好？
2. **混合检索策略：** grep + 向量检索的组合是否效果最佳？
3. **实验 2 的具体结果：** 在干扰环境下，grep 和向量检索的鲁棒性到底如何？
4. **成本权衡：** grep 可能需要更多上下文窗口，向量检索可能更节省 token，但效果更差。成本-效果曲线如何？
5. **Agent 长尾场景：** LongMemEval 的 116 个问题是否覆盖了真实世界的复杂 agent 任务？

---

## 结论

这篇论文的价值不在于证明"grep 永远比向量检索好"，而在于：

- **提醒我们：agent 搜索系统的评估需要在真实 agent 环境中进行**，harness 和工具调用范式是关键变量。
- **打破 RAG 的默认假设**：向量检索不是唯一的答案，在代码等结构化数据场景下，传统文本搜索可能更有效。
- **为 agent 工程提供了实用的测试框架**：你可以用类似的方法评估你自己的 agent 检索策略。

对于正在构建 LLM agent 的工程师来说，这篇论文的核心建议是：**先测试 grep，再考虑向量检索，并且在真实 agent 环境中做对比，而不是只看离线检索指标。**
