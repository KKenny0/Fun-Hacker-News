# 验证债务：AI 生成代码的隐藏成本

> **字数**：约 1000 字
> **风格**：平衡务实、基于实践

---

AI coding agent 在软件工程圈引发了明显分化：

**支持者**：离不开至少两个 agent，不上厕所都在烧 token，否则"不高效"。  
**怀疑者**：AI 反而拖慢他们，自己动手更快。

**两者都对。**

当前实际体验：agent 10 分钟产出 impressive diff，你花 1 小时确认它没漏掉什么。上下文蒸发，20 万 token 听起来慷慨，直到 agent 开始压缩对话，10 分钟前的约定都忘了。输出冗长，你要聚焦的改动，却得到一篇论文。工具集成好坏参半。

但有些东西变了。没人再争论"是否可行"，而是在争论"如何做"。

## 核心概念：验证债务

**什么是验证债务？**

验证债务（verification debt）——生成速度与验证速度之间的鸿沟。每当你点击批准一个你没有完全理解的 diff，你就在向未来借贷。

**与技术债务的区别：**
- **技术债务**：通过摩擦显现——慢构建、tangled 依赖、触碰某模块的恐惧
- **验证债务**：滋生虚假信心——代码库看起来干净，测试绿色，但六个月后发现你构建的正是规格说的，却不是客户真正想要的

Agent 可以在几分钟内生成看起来合理的 diff。测试通过，commit message 比一半人类写的还好。但陷阱在于：**"看起来对"不等于"就是对的"**。

## 从"Party Trick"到生产可用

**几年前**：ChatGPT 是"盒子里的脑"——强大推理，零连接。就像 iPhone 没有网络——impressive 演示，对真实工作无用。

**去年**：工具开始追赶，自动补全让位给 agent 工作流，但仍然笨拙。

**现在**：几件事同时发生——模型足够好（Opus 4.5、GPT-5），产品成熟（terminal-native agents），用户更有经验（prompting 是技能）。不是单一突破，是更好的模型、工具、用户同时到达。

**真正的转折**：当作者连接 Claude Code 到 Snowflake 数据仓库，一个帮助写 SQL 的工具变成了全职分析师——自己爬取 schema，交叉引用代码和 Confluence，带回没想到要找的洞察。不是"AI 为我写代码"，而是"AI 可以在真实世界中行动"。

## 人类瓶颈

**不舒服的真相：**

如果 AI 让每个工程师生产力提升 50%，组织不会得到 50% 更多产出。会得到 50% 更多 pull request、50% 更多文档、50% 更多设计提案——**有人必须审查所有这些**。

当每个人都这样做，审查成为约束。瓶颈向上游移动：决定构建什么、定义"完成"、理解领域、做风险和权衡的判断。

**认知负荷转化：**

没人想审查 AI slop。但发件箱堆满了急切 agent 的输出，快过蹚过它们。

软件工程一直是知识工作——分析、分享上下文、构建共享理解。AI 可以帮你更快找到信息，但你仍然必须理解它。AI 不移除认知负荷，它转化它。

**停止思考的恐惧：**

担心不是关于工作——而是我们会停止思考。这和 Google 出现时同样的恐惧。实际发生了什么：我们停止记忆 API 签名，开始解决更难的问题。技能转移了，但没有缩小。AI 是同样的模式，提升一个层次。

## 验证清单

如何避免验证债务累积？实用清单：

1. **Agent 实现的是正确逻辑，还是忠实编码有缺陷的规格？**（它不会质疑你的意图）
2. **Agent 对领域做了什么假设？**
3. **此改动引入了什么权限、数据访问或副作用？**
4. **你敢以你的名字担保这做了用户实际需要的——不只是工单说的？**

如果最后一个答案是"可能"，你还没完成审查。
