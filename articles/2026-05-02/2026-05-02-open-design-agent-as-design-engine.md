# Open Design：当编码代理成为设计引擎

2026年4月17日，Anthropic发布了**Claude Design**——一个让LLM不再写散文、而是开始交付设计制品的产品。它迅速走红，但同时也暴露了一个关键问题：完全封闭——闭源、付费、云托管、锁定在Anthropic的模型和技能体系上。没有自托管，没有Vercel部署，没有替换代理的能力。

5月2日，一个名为**Open Design**的项目在Hacker News上获得了164颗星，给出了这个问题的开源答案。

## 产品的核心假设

Open Design的假设很简单：**最强的编码代理已经在你的笔记本电脑上了**。

它不打包一个新的代理。相反，它扫描你的`PATH`，自动检测12种主流编码代理CLI：Claude Code、Codex CLI、Devin for Terminal、Cursor Agent、Gemini CLI、OpenCode、Qwen Code、GitHub Copilot CLI、Hermes（ACP）、Kimi CLI（ACP）、Pi（RPC）、Kiro CLI。

检测到哪一个，哪一个就成为候选的设计引擎——通过stdio驱动，每个CLI一个适配器，可以在模型选择器中一键切换。

## 技术架构的三层设计

Open Design采用了一个清晰的分层架构：

**Web层（Next.js 16）**：聊天界面、文件工作区、iframe预览、设置和导入功能。

**本地守护进程（Express + SQLite）**：这是唯一的服务器层，负责代理检测、技能注册、设计系统目录、项目/会话/消息持久化、以及OpenAI兼容的BYOK代理。

**代理层**：守护进程通过`child_process.spawn`启动CLI，将工作目录设置为项目的`.od/projects/<id>/`文件夹。代理获得真实的`Read`、`Write`、`Bash`、`WebFetch`工具，操作真实的文件系统。

## 六个负载支撑的设计理念

Open Design的价值主张建立在六个核心思想之上：

### 1. 代理是你现有的，不是他们打包的

守护进程启动时扫描`PATH`，找到的任何编码代理CLI都自动成为候选设计引擎。没有安装CLI？`POST /api/proxy/stream`提供了相同的管道——只需粘贴任何OpenAI兼容的`baseUrl` + `apiKey`，守护进程就会转发SSE块，并在边缘拒绝回环/链路本地/RFC1918目标以防止SSRF。

### 2. 技能是文件，不是插件

遵循Claude Code的`SKILL.md`约定，每个技能就是`SKILL.md` + `assets/` + `references/`。将文件夹放入`skills/`，重启守护进程，它就会出现在选择器中。这意味着技能的添加、修改、共享都像文件操作一样简单。

### 3. 设计系统是可移植的Markdown，不是主题JSON

采用来自`awesome-design-md`的9部分`DESIGN.md`模式——颜色、排版、间距、布局、组件、动效、声音、品牌、反模式。每个制品都从激活的系统中读取。切换系统→下一次渲染使用新的token。

内置了72个设计系统，涵盖Linear、Stripe、Vercel、Airbnb、Tesla、Notion、Apple、Anthropic、Cursor、Supabase等主流产品，以及57个来自`awesome-design-skills`的设计技能。

### 4. 交互式问题表单防止80%的返工

OD的提示栈硬编码了一个`RULE 1`：每个新的设计简报都以`<question-form id="discovery">`开始，而不是代码。表面·受众·语气·品牌语境·规模·约束。一个长的简报仍然会留下设计决策开放——视觉语气、色彩立场、比例——而这些正是表单在30秒内锁定的东西。

这是从`huashu-design`中提炼出的**初级设计师模式**：提前批处理问题，早期展示可见的内容（即使是有灰色块的线框图），让用户可以廉价地重定向。

### 5. 守护进程让代理感觉像在你的笔记本电脑上，因为确实如此

守护进程使用`cwd`设置为`.od/projects/<id>/`下的项目制品文件夹来启动CLI。代理获得`Read`、`Write`、`Bash`、`WebFetch`——针对真实文件系统的真实工具。它可以`Read`技能的`assets/template.html`，`grep`你的CSS以获取十六进制值，写入`brand-spec.md`，放置生成的图像，并生成`.pptx` / `.zip` / `.pdf`文件，这些文件在回合结束时作为下载芯片出现在文件工作区中。

### 6. 提示栈就是产品

你在发送时组合的内容不是"系统+用户"，而是：

```
发现指令（第1回合表单、第2回合品牌分支、TodoWrite、5维批评）
  + 身份宪章（OFFICIAL_DESIGNER_PROMPT、反AI-slop、初级通过）
  + 活跃的DESIGN.md（72个系统可用）
  + 活跃的SKILL.md（31个技能可用）
  + 项目元数据（类型、保真度、演讲者注释、动画、灵感id）
  + 技能侧文件（自动注入的预检：读取assets/template.html + references/*.md）
  +（牌组类型，无技能种子）DECK_FRAMEWORK_DIRECTIVE（导航 / 计数器 / 滚动 / 打印）
```

每一层都是可组合的。每一层都是你可以编辑的文件。

## 反AI-slop的机制设计

Open Design继承了`huashu-design`的完整机制，并将其纳入OD的提示栈中，使其通过每个技能的侧文件预检可执行：

- **问题表单优先**：第1回合只有`<question-form>`——没有思考，没有工具，没有叙述。用户以无线电速度选择默认值。

- **品牌规范提取**：当用户附加截图或URL时，代理在编写CSS之前运行一个五步协议（定位·下载·grep十六进制·编码`brand-spec.md`·声张）。**从不从记忆中猜测品牌颜色。**

- **5维批评**：在发出`<artifact>`之前，代理默默地在哲学/层次/执行/特异性/克制五个维度上给自己的输出打分1-5。任何低于3/5的都是回归——修复并重新评分。两次通过是正常的。

- **P0/P1/P2检查表**：每个技能都附带一个`references/checklist.md`，其中包含硬P0门槛。代理必须在发出之前通过P0。

- **Slop黑名单**：激进的紫色渐变、通用的emoji图标、带有左边框强调的圆角卡片、手绘SVG人类、Inter作为*显示*字体、虚构的指标——在提示中明确禁止。

- **诚实的占位符>虚假的统计**：当代理没有真实数字时，它写入`—`或标记的灰色块，而不是"10倍更快"。

## 为什么这很重要

Open Design的意义不仅在于它是一个Claude Design的克隆品。它提出了一种不同的愿景：

**BYOK（Bring Your Own Key）的极致形式**：不仅是Bring Your Own API Key，更是Bring Your Own Agent。你已经在为Claude Code、Cursor Agent、Copilot CLI付费——Open Design让你利用这些投资，而不是要求你再订阅一个服务。

**技能的文件级抽象**：技能不是需要编译和安装的插件。它们是文件夹。这意味着Git工作流、PR审查、社区贡献都变得透明。

**设计系统作为可移植资产**：`DESIGN.md`模式让设计系统可以在项目、团队、甚至不同工具之间移植。切换设计系统不是重写CSS，而是切换一个文件。

**质量控制的硬约束**：问题表单、5维批评、P0/P1/P2检查表、slop黑名单——这些不是可选的最佳实践，而是硬编码的规则。代理无法绕过它们。

## 潜在限制与挑战

Open Design并非没有挑战：

- **学习曲线**：用户需要了解不同的编码代理CLI、技能系统、设计系统模式。这不是一个"开箱即用"的工具。

- **依赖外部工具**：它依赖于你的PATH上有可用的编码代理CLI。如果没有，你只能使用BYOK代理，但这会失去一些本地文件系统集成的优势。

- **用户界面仍在演进**：README明确指出这是早期实现。评论模式外科编辑、AI发出的调整面板UX等功能仍在进行中。

- **文档和社区**：作为开源项目，它需要文档、教程、示例和活跃的社区才能达到Claude Design的易用性水平。

## 评估：有潜力的实验，但需要更多时间

Open Design是一个雄心勃勃的实验。它试图在保持开放性的同时，复制Claude Design的核心价值主张：让AI成为一个有文件系统、确定性调色板和清单文化的高级设计师。

它的优势很清晰：真正的BYOK、文件级技能抽象、便携的设计系统、硬编码的质量控制。但它的劣势也很明显：更陡峭的学习曲线、对现有工具的依赖、仍在演进的UI。

如果你已经有一个编码代理CLI在工作，并且想要一个可以自我托管、可定制、不锁定的设计工作流，Open Design值得一看。如果你只是想要一个"让AI为我设计东西"的按钮，Claude Design可能仍然是更好的选择——至少目前是这样。

但Open Design的真正价值可能不在于它是否能在短期内与Claude Design竞争，而在于它提出的问题：当最强的编码代理已经在你的笔记本电脑上时，为什么要再打包一个？当技能可以是文件时，为什么要让它们成为插件？当设计系统可以是Markdown时，为什么要让它们成为专有主题？

这些问题的答案可能会重新定义我们思考AI设计工具的方式。
