# 微软弃用 Claude Code 的战略选择

微软计划在 6 月底前取消内部大部分 Claude Code 许可证，推动开发者转向 GitHub Copilot CLI。这一决定不仅涉及微软自有产品的战略重塑，也折射出大型科技公司内部工具选型与产品竞合之间的复杂博弈。

## 事件经过

根据 The Verge 报道，微软从 2025 年 12 月开始向内部开发者开放 Claude Code 访问权限，让数千名员工日常使用，特别是吸引项目经理、设计师等非技术背景员工首次尝试编码。Claude Code "proved very popular"。

然而微软现在准备收回这项福利。Experiences + Devices 团队（负责 Windows、Microsoft 365、Outlook、Teams、Surface 的工程师）将在 6 月底前停止使用 Claude Code。公司告知员工将在未来几周内开始将工作流迁移到 GitHub Copilot CLI。

微软体验与设备集团执行副总裁 Rajesh Jha 在内部备忘录中解释："我们的目标是通过两种工具快速学习，在真实工程工作流中基准测试，并理解什么最能支持我们的团队。"

## 微软的真实动机

微软官方将这一决定描述为工具收敛。但信源透露这也是一个财务决策。6 月 30 日是微软当前财年的最后一天，取消 Claude Code 许可证是"an easy way to cut some operating expenses"。

更深层的动机在于产品战略。Claude Code 的流行实际上损害了微软自家的 GitHub Copilot CLI —— 一个运行在 Visual Studio Code 等 IDE 之外的命令行版本。报告指出："it has also undermined Microsoft's new GitHub Copilot CLI coding tool"。

微软希望将 Copilot CLI 打造为"main agentic command line interface tool"，并与 GitHub 深度合作定制产品。这符合大型科技公司通过内部使用来打磨自有产品的策略。

值得注意的是，微软与 Anthropic 的其他合作并未受到影响。Claude 模型仍可通过 Copilot CLI 访问，且在 Microsoft 365 Copilot 中继续使用。这表明微软的决策是工具层面的战略选择，而非供应商层面的全面切割。

## 行业影响

微软自家开发者更偏爱 Claude Code 而非 GitHub Copilot CLI，这一事实揭示了 AI 编程工具市场的竞争现状。产品性能和用户体验仍在快速演进，用户偏好可能会快速转变。

这一事件也反映了科技巨头在 AI 时代的两难：需要获取最佳工具提升内部效率，同时又要投资自研产品、控制成本、避免对第三方形成依赖。微软曾考虑收购 Cursor 来缩小差距，但"has started looking at different AI startups to bolster its AI ambitions and avoid potential regulatory scrutiny"。

当前行业环境下，大型科技公司通过收购来补齐产品短板的路径越来越受限。内部投入和战略合作成为主要选项，这也意味着产品迭代速度和用户体验可能面临更大压力。

微软现在需要向 GitHub 团队施压，要求其快速改进 Copilot CLI，使其超越 Claude Code。报告指出："the pressure is now on Microsoft's GitHub team to improve Copilot CLI and try to surpass Claude Code in the process"。

这不仅是微软的挑战，也是整个 AI 编程工具行业的问题。如何在快速发展的技术环境中平衡内部需求、产品战略和市场竞争，将决定谁是最终的赢家。

微软这一决策标志着 AI 工具竞争进入新阶段：不再是简单购买最佳解决方案，而是需要构建可持续的内部能力。工具选择变得复杂，但战略清醒从未如此重要。
