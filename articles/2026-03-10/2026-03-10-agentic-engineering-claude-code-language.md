# 当 AI 写下每一行代码：一位开发者用 Claude Code 构建编程语言的实验

> 用四周时间、让 Claude 生成所有代码、自己一行都没读——这听起来像是在"偷懒"，但作者 Ankur Sethi 用这套方法真的做出了一个能运行的编程语言 Cutlet。更关键的是，他总结出了一套"代理工程学"的方法论。

## 摘要

前端工程师 Ankur Sethi 用 Claude Code 在四周内从零构建了一个名为 Cutlet 的动态编程语言。整个过程中，他让 Claude 生成每一行代码，自己甚至没有阅读过代码本身。取而代之的是，他构建了一套完整的"护栏系统"——测试套件、示例程序、静态分析工具、内存安全检测——让 AI 在这个安全网内自主工作。这篇文章记录了他的方法论、踩过的坑，以及对"代理工程学"的深刻反思。

---

## 一、为什么是编程语言？

Sethi 是一名前端工程师，此前尝试过用 LLM 构建网页应用，但屡屡碰壁。他的观察是：

> **LLM 擅长写复杂业务逻辑，但不擅长任何需要视觉设计能力的任务。**

用自然语言描述响应式布局和动画几乎不可能。他曾花数小时和 Claude 纠结布局问题——Claude 信誓旦旦说修好了，但他肉眼可见问题依旧存在。

那什么样的项目适合"全权交给 AI"？他总结了两个判断标准：

1. **能否用自动化方式定义和验证成功标准？**
2. **这个问题是否在 LLM 训练数据中出现过（或类似问题）？**

如果两个答案都是"是"，那就可以尝试代理工程。编程语言完美符合：
- 有大量现有实现、论文、CS 教材可供参考
- 可以用确定性测试验证正确性（输入程序 → 预期输出）
- 纯命令行环境，无需截图、浏览器、API 调用

---

## 二、"瀑布式"工作流：先想清楚，再放手

Sethi 发现，和 LLM 协作的最佳模式不是"边写边改"，而是**把所有思考前置**。他的工作流：

1. **讨论阶段**：和 Claude 讨论新功能（比如循环）、参考其他语言实现、设计考量
2. **制定计划**：让 Claude 生成详细的实现计划，拆分成小步骤
3. **审查计划**：反复打磨，覆盖边界情况、潜在陷阱
4. **写入文档**：把计划存入 `plans/doing/` 目录，作为原子化的回滚单元
5. **放手执行**：在 Docker 容器中启动 Claude，给完全权限，让它自主实现

这种"先规划后执行"的模式和他以往的开发习惯完全不同。他习惯在编码过程中有机地发现问题形态，但 LLM 不擅长这种迭代式、探索式的开发——它们更擅长"一次性大改动"。

---

## 三、为 AI 构建工作环境

这是全文最精彩的部分。Sethi 认为：**AI 是计算机程序，它对世界的认知仅限于它能访问的代码目录。** 要让它高效工作，必须主动给它"眼睛"和"工具"。

### 3.1 测试套件
项目指令要求 Claude 在写任何新代码前先写测试，并确保测试失败。每次重大改动后都要运行测试。不断增长的测试套件能快速发现回归问题，同时也充当文档和规格说明。

### 3.2 示例程序
`examples/` 目录存放了大量示例程序（大部分由 Claude 自己写），既是文档，也是端到端测试。每次代码改动后都要运行所有示例并验证输出。

### 3.3 静态分析
Cutlet 使用 clang-tidy 和 clang-format 确保代码质量基线。有趣的是，clang-tidy 经常产生诊断信息，迫使 Claude 重写部分代码。

### 3.4 内存安全检测
`make test-sanitize` 用 AddressSanitizer、UndefinedBehaviorSanitizer、LeakSanitizer 重新编译整个项目并运行测试。这能捕获测试和 linter 都发现不了的内存错误（use-after-free、缓冲区溢出、未定义行为）。

### 3.5 运行时内省工具
让 Cutlet 能够在执行前 dump 出 token 流、AST、字节码。这样 AI 无需阅读源码或调试器就能快速判断执行管道中哪里出了问题。

### 3.6 管道追踪脚本
一个 Python 脚本，把 Cutlet 程序喂给解释器并捕获完整编译管道（token 流、AST、字节码反汇编），然后映射回源码位置。当 AI 需要添加新语言特性时，可以先在类似特性上跑这个脚本，精确知道要碰哪些文件和函数。

### 3.7 完全权限
Sethi 在 Docker 容器中运行 Claude，启用 `--dangerously-skip-permissions`，给完全 sudo 权限。他认为这是在大型项目中使用编程代理的唯一可行方式——手动确认权限提示在认知上非常消耗，而限制代理的能力会降低它们的效率。

## 四、优化代理循环

Claude 有时会做低效操作：
- 运行命令后发现输出太长，又用 `head -n 10` 重跑
- 运行 `make check` 忘了 grep 错误，又跑一次捕获输出

Sethi 的做法是观察这些重复模式，把它们封装成脚本。虽然很手动、很枯燥，但能显著加速代理循环。

他希望未来的 Claude Code 能自动分析自己的工具调用输出，建议用户写哪些脚本。

---

## 五、软件工程死了吗？

Sethi 的答案是：**没有**。

他承认，如果没在大学上过编译器课程、没读过《Crafting Interpreters》，他做不出 Cutlet。他仍然需要做技术决策，而这些决策需要领域知识和经验。

但他也学到了一堆新技能——一种"奇怪、新颖、不同"的技术知识。

> 在那些永远不会碰 LLM 的人的领域，LLM 可能永远无法渗透。他的朋友在做底层多媒体系统，用 LLM 的效果远不如做 Web 应用的朋友。这种情况可能持续很多年。

---

## 六、一个坦诚的心理健康警告

这是全文最打动我的部分：

> 很容易对代理工程工具上瘾。做这个项目时，我经常半夜坐在电脑前说"再一个 prompt"，仿佛在玩世界上最冷门的《文明》游戏。

更成瘾的是这些工具的**不可预测性**。扔一个问题给 Claude，你永远不知道它会交出什么——可能一枪解决你卡了几周的难题，也可能搞出一团糟。就像老虎机，你永远不知道下一把会发生什么。**而庄家（Anthropic）永远赢。**

Sethi 现在给自己设定了使用 Claude 的时长和频率限制。他担心：我们作为一个社会，在社交媒体上已经全面失败了，LLM 很可能重演那个剧本。

---

## 七、这算谁的作品？

Sethi 很诚实地面对这个问题：

> 我不把这视为"我的"作品。感觉不对。也许未来我的感受会变，但我现在没法说服自己。

因为这个实验之所以成立，是因为 Claude 的训练数据里有无数程序员、学者、作家无偿贡献的工作成果。Cutlet 几乎肯定包含了 Lua 和 Python 解释器的代码——Claude 在讨论语言特性时经常引用它们，他也亲眼看到大量《Crafting Interpreters》的代码混入代码库。

所以他没有给 Cutlet 的 GitHub 仓库加任何许可证。

> Cutlet 属于所有在互联网上发布过作品的编程语言设计者、实现者和教育者的集体意识。

---

## 分析：为什么这篇文章重要？

这不是另一篇"我用 AI 写了个 Hello World"的浅尝辄止。Sethi 做了一次严肃的、有方法论支撑的实验，并诚实地记录了成功、困惑、和心理代价。

**几个关键洞察：**

1. **代理工程不是"甩手掌柜"**：它需要更精心的规划、更完善的测试基础设施、更深刻的领域知识。你从"写代码的人"变成了"设计系统和流程的人"。

2. **护栏比代码更重要**：Sethi 把精力花在构建测试、lint、sanitizer、调试工具上，而不是写代码。这些基础设施的价值远超任何单次代码生成。

3. **新的技能树**：理解哪些问题适合 LLM、如何用自然语言精确表达意图、如何为 AI 创建工作环境、如何优化代理循环——这些是"代理工程学"的核心技能。

4. **心理成本被低估了**：我们谈论 AI 效率时很少谈论成瘾风险。不可预测的奖励机制是成瘾的经典模式，而 AI 恰好提供了这种体验。

5. **所有权问题是真实的**：当你的代码 99% 来自 AI，而 AI 的训练数据来自全人类的集体贡献，"谁拥有这段代码"不再是一个简单的问题。

---

## 结语

Cutlet 项目是一个 proof of concept——"我能把 Claude Code 推到多远"。Sethi 目前在找前端工程师的工作，可能没时间继续维护这个项目。但他开放了所有源码、计划文档、脚本，为想尝试"代理工程"的人提供了一份珍贵的蓝图。

这不是"软件工程已死"的宣言，而是"软件工程正在演变"的实地报告。变的是工具和技能，不变的是对规划、质量、责任的追求。

至于心理健康的部分——也许我们该开始认真讨论了。
