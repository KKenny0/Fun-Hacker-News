## 事件背景

2026 年 3 月 31 日，Anthropic 在一次常规软件更新中意外泄露了 51.2 万行 Claude Code 的源代码——只因缺失了一个配置文件。在日出之前，代码库就被镜像到了 GitHub；早餐前，有开发者用 AI 工具将其重写为 Python 版本，"claw-code" 仓库单日获得 10 万星，创下 GitHub 历史最快记录。

随之而来的是 DMCA 下架通知，然后是一个没人能给出清晰答案的问题：

> 如果 Claude Code 如 Anthropic 首席工程师承认的那样，主要由 Claude 自己编写，Anthropic 是否真正拥有它？你能对版权法可能无法保护的代码发出 DMCA 下架吗？

这个事件将所有关于 AI 生成代码所有权的未决问题压缩到了一个新闻周期里。而这些问题同样适用于你的代码库。

## 核心法律框架：AI 生成代码版权的三个维度

### 1. 人为创作性（Human Authorship）

**法律底线**：版权只保护人类创作的作品。

美国版权局一贯确认这一点，DC 巡回法院在 Thaler 案中予以维持。当最高法院在 2026 年 3 月拒绝审理 Thaler 上诉时，它既未认可下级法院的推理，也未在全国范围内解决这一问题。拒绝听证意味着法院选择不审理此案，仅此而已。

但更重要的是：AI 生成且未经有意义人类修改的代码，可能无法被任何人版权所有。如果竞争对手复制它，你可能没有法律追索权，因为这段代码实际上处于公有领域。

"有意义的人类作者身份"是决定你的代码是否受保护的关键词。版权局故意拒绝用百分比或编辑次数来量化它，因为法院寻找的是证据——证明人类做出了真正的创造性决策：
- 选择架构
- 决定拒绝什么
- 重构输出以适应特定设计

在代理式工作流中，这个区分比看起来更难确立。想想典型的 Claude Code 会话：
- 你写一行提示词："为 API 构建速率限制模块"
- Claude Code 规划方法，生成五个文件，迭代三个版本
- 你审查输出，运行测试，合并

在这个序列中，你的贡献是你的架构意图和最终批准。这是否构成法庭上的"有意义的人类作者身份"是一个未决问题，目前还没有明确的法院裁决。

诚实的答案是：对于你大量重定向的模块，可能是；对于你原样接受的代码，可能不是；对于中间的所有情况，不清楚。

### 2. 工作雇佣原则（Work-for-Hire Doctrine）

在思考你的代码是否可版权之前，有一个更紧迫的问题：即使可版权，它真的是你的吗？

你的雇佣合同几乎肯定规定你在工作时构建的任何东西都属于你的雇主。版权法中这个原则有个名称：**工作雇佣原则**。根据该原则，员工在雇佣范围内创建的任何代码由雇主拥有，雇主被视为法律作者，无论代码是手写、由 Claude Code 生成，还是两者结合。

在工作时间、工作项目、工作机器上使用 AI 编程工具，不会改变结果的所有权。

大多数雇佣合同走得比原则的默认设置更远。查找合同中名为"知识产权"、"IP 转让"或"工作成果"的部分。打开合同，搜索这些术语，阅读该部分。说以下任何内容的条款几乎肯定覆盖你的 AI 辅助代码：
- "使用公司设备或资源创建的任何工作成果"
- "受雇期间做出的任何发明或开发"
- "使用公司许可工具的辅助创建的任何软件"

**第三个是需要关注的。** 如果你的雇主为团队授权 Claude Code、Cursor 或 Copilot，而你使用相同的工具构建副业项目，广泛的 IP 转让条款可能会让雇主对该项目拥有主张权，即使你是在业余时间构建的。

旧金山的一位高级开发人员今年早些时候描述了完全这种情况。他将 Claude Code 用于工作项目，以及在晚上和周末构建的个人健身追踪应用。他的公司更新了其 IP 政策，并声称他使用 AI 辅助构建的一切，包括个人应用，理由是因为 Claude 访问了工作 IDE 中的开放文件，任何 AI 输出都是公司 IP 的衍生作品。

这是这种情况可以拉伸到多远的最清楚例子。他公司的说法依赖于一个短语：AI 工具对公司的代码库是"上下文感知"的。这个论点在法律上站不住脚，因为 IDE 中的上下文可见性不会使 AI 输出成为附近打开的文件的衍生作品，而 Claude 能看到的内容和它生成的内容之间的联系是概率模式补全，而不是复制。但这个论点说明了雇主开始主张的内容。如果条款足够广泛，无论 AI 实际做了什么，它都有表面有效性。

**实用规则**：如果你在构建副业，使用个人账户、个人机器和你自己支付的工具。完全将你雇主授权的工具排除在该工作流之外。

### 3. 开源污染（Open Source Contamination）

即使你拥有 AI 生成的代码，你可能已经用你无法看到的开源许可证污染了它。

AI 编程工具在海量的公共代码上训练，包括 GPL、LGPL 和其他 copyleft 许可证授权的代码。**Copyleft 许可证携带特定义务，随代码同行**：
- 如果你分发是 GPL 授权代码衍生的软件，你必须以相同许可证发布你自己的源代码
- 这即使在你不知道你合并的代码是 GPL 授权的情况下也适用
- "我不知道"不是 copyleft 违反的辩护

当 AI 工具从其训练数据中复制 GPL 授权代码的大量逐字部分，而你在不发布源代码的情况下将该代码发布在商业产品中时，你可能已经造成了 copyleft 违反，甚至从未接触过原始仓库。

侵权的法律标准是大量逐字复制，而不是功能相似性或类似性，这个区别很重要：生成像 GPL 代码一样工作的 AI 工具，与逐字复制 GPL 代码的 AI 工具是不同的。风险位于该谱系的逐字末端，问题是你无法在不运行扫描的情况下知道你的代码库位于线的哪一侧。

2026 年初的 chardet 社区争议使这一点具体化。这不是提起的诉讼，而是开源社区内的公开争议，提出问题但未在法律上解决。一位开发人员使用 Claude 重写 chardet（一个 Python 字符编码库），并以 MIT 许可证重新发布，理由是 AI 重写是免费的"净室"实现，免于原始 LGPL 许可证。

**社区争论的法律问题**：如果 Claude 是在 LGPL 授权的代码库上训练的，其输出复制该代码的大量逐字部分，输出能否被视为无许可证？chardet 争议没有清晰解决，法院也没有对这一具体问题发出明确裁决。已确定的是，无论如何产生，逐字复制 GPL 代码都违反许可证。未确定的是，复制训练数据模式的 AI 生成输出是否算作逐字复制。在并购中为公司提供建议的律师的工作假设是可能如此，该假设现在正作为收购尽职调查中的标准条件出现。

## 实践建议：四项具体行动

### 1. 对 AI 辅助代码库运行许可证扫描

做得好的工具：
- **FOSSA** — 最全面，企业广泛使用
- **Snyk Open Source** — 适合开发团队工作流，与 GitHub 集成
- **Black Duck** — 并购尽职调查标准

每个工具都会扫描你的代码库，标记匹配已知开源库的代码，并识别附带的许可证。如果你发布商业产品且从未运行过其中任何一个，你在假设上运行。扫描需要一个下午，成本低于版权争议的第一个小时。

### 2. 随时记录你的人为创造性贡献

确立有意义的人类作者身份的证据与你在正常工程工作流中已经产生的证据相同。你只是需要有意保留它，而不是让它消失。

需要保留的内容：
- 描述你改变了什么以及为什么的提交信息，而不仅仅是 AI 生成了什么。"重构 Claude 的模块架构，拒绝初始状态管理方法，从头重写错误处理"是证据。"添加速率限制模块"不是。
- 提示词日志。Claude Code 和 Cursor 都保留交互历史。导出或截屏你做出重大架构决策的会话。
- 设计文档、ADR，或在生成代码之前存在的任何笔记，显示你在 AI 构建之前指定了结构。

第二个提交信息与第一个的区别，是可辩护的作者权主张与清晰的"Claude 写了这个"记录之间的差异。

### 3. 在构建任何副业之前阅读雇佣合同中的 IP 条款

打开合同，搜索"知识产权"、"IP 转让"或"工作成果"，仔细阅读该部分。具体语言决定你的风险敞口：
- "雇佣期间创建的工作成果"比"使用公司资源创建的工作成果"更窄
- "与公司业务相关"比"任何软件开发"更窄
- "公司许可工具"是捕获副业项目中的 AI 编程工具的短语

如果条款广泛且你想独立构建某物，你有三个现实选择：在开始前协商书面豁免（在新角色开始时比在受雇期间更容易），在个人时间在个人机器上完全使用个人工具，或者接受主张存在并决定风险是否值得。

### 4. 在商业用途发布前检查你使用的 Anthropic 计划

访问 [anthropic.com/legal](https://www.anthropic.com/legal) 并比较消费者条款与商业条款。重要的区别：
- **消费者条款（免费和专业计划）**：Anthropic 将输出分配给你，但 IP 赔偿范围更窄，覆盖的场景更少。
- **商业条款（API 和企业）**：Anthropic 将输出分配给你，并将为你辩护针对从服务授权使用及其输出中产生的版权侵权主张。

如果你使用免费或专业计划在商业产品中发布 AI 辅助代码，赔偿差距是真实的。API 或企业协议是适当的层级。注意，两者都不赔偿你代码库中许可证污染导致的下游 GPL 违反。那是你用行动 1 中的许可证扫描解决的治理问题。

## 关键认知缺口与待解决问题

### 已确立的法律（三项）：
- 缺乏人类作者身份的作品不可版权
- 工作雇佣原则无论代码如何生成都适用
- 逐字复制 GPL 授权代码违反许可证

### 新兴共识但无明确法院裁决（两项）：
- 在代理式工作流中多少人类指导足以确立有意义作者身份
- 复制训练数据模式的 AI 输出是否算作逐字复制

### 真正的推测（一项）：
- 任何这些是否会在近期大规模诉讼

大多数代码版权主张从未到达法院。**未决问题今天变得具体的地方是并购尽职调查和机构融资，收购方和投资者已经将这些问题作为关闭的条件提出。**

**如果这两者目前都不适用于你的情况，上述四项行动仍然值得做，但紧迫性低于本文可能暗示的程度。**

## 进一步阅读推荐

1. **美国版权局 — 版权与人工智能（第二部分：可版权性）**
   - AI 辅助作品中什么构成有意义人类作者身份的主要监管来源。第二部分涵盖办公室在审查 AI 生成内容注册时应用的具体测试。如果你想确切知道法律线在哪里，这是必不可少的。

2. **Andersen v. Stability AI, Midjourney, DeviantArt — 第九巡回法院案卷**
   - 关于 AI 训练数据和版权侵权的奠基案例，目前正在塑造法院对 AI 模型学习和复制内容的思考。与 GPL 污染问题相关，大多数开发者尚未将其联系起来。

3. **Doe v. GitHub, Inc. — 第九巡回法院上诉**
   - 关于 Copilot 是否未经许可复制授权代码的诉讼中的争议。跟踪这个：第九巡回法院的决定将设定确定 AI 生成代码携带开源模式是否构成版权侵权的标准。

4. **GitHub — Copilot 和版权：你需要知道的**
   - GitHub 关于为什么 Copilot 输出不侵权的自身法律立场。作为反论点值得阅读：理解他们提出的论点有助于你理解它在哪里强有力，在哪里有局限性，特别是在 GPL 训练数据问题上。

5. **FOSSA — 理解开源许可证义务**
   - 开发人员友好的参考，关于 copyleft 义务在实践中如何工作：什么触发源披露要求，什么构成衍生作品，以及 GPL、LGPL 和 AGPL 在其覆盖范围上如何不同。该主题上可用的最清晰的平白语言指南。

## 分析总结

这篇文章的价值在于它清晰地拆解了 AI 生成代码版权的三个独立维度，避免了将问题混为一谈。更关键的是，它提供了可操作的实践建议——从许可证扫描到文档记录，从合同审查到服务条款选择。

对于开发者而言，最需要警惕的可能是"上下文感知"这个模糊概念被雇主泛化解释的风险。案例中的健身追踪应用事件表明，企业已经开始尝试将 AI 工具的上下文能力与衍生作品概念绑定，这个法律论点虽站不住脚，但在合同层面仍可能造成实际困扰。

另一个值得关注的趋势是，法律争议的主要战场不是法庭，而是并购尽职调查。这意味着即使不面临诉讼，AI 辅助代码的版权问题也会在融资和收购交易中成为实质性障碍。FOSSA、Snyk、Black Duck 等工具因此成为标准配置，而不仅仅是"可选的安全措施"。

从法律发展轨迹看，Thaler 案确立了"无人类作者即无版权"的底线，但未解决"多少人类参与足够"的量化问题。Allen v. Perlmutter 案（600+ 详细提示词 + Photoshop 编辑仍被拒注册）显示版权局的门槛相当高。这对开发者的启示是：如果希望代码受保护，需要在设计决策层面留下明确的创造性证据，而不仅仅是验收测试和合并提交。

GPL 污染问题的不确定性最大。chardet 争议未决，Doe v. GitHub 仍在进行，"AI 模式复制是否等于逐字复制"尚无法律定论。但并购市场的实际行动表明，企业在风险偏好上采取了"可能即视为"的保守立场。

最终，这篇文章的核心信息是：无论法律如何演进，主动的文档记录、工具隔离和许可证扫描都是成本最低的风险管理手段。法律可能需要数年才尘埃落定，但开发者的职业资产保护不能等待那个时刻。
