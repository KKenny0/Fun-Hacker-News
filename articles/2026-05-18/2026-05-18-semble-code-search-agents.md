# 让编程 Agent 不再"暴力搜索"：Semble 如何将代码搜索的 Token 消耗降低 98%

## 一、问题：编程 Agent 的 Token 黑洞

当 AI 编程助手在陌生代码库中寻找特定功能时，它们通常会采用一个非常"笨"的方法：先用 grep 搜索关键词，然后把匹配的整个文件读进上下文。

这种策略的缺陷显而易见：为了找到几个关键函数，Agent 往往需要下载几千行无关代码。假设你在寻找"如何处理用户认证"，一个 2000 行的 `auth.py` 文件可能只包含 100 行相关代码，但 Agent 不得不把整个文件都读进来。

这不仅浪费昂贵的 API token，还会挤占上下文窗口，迫使 Agent 进行多轮对话或在一次对话中只能处理更少的信息。对于使用长上下文模型（如 128k token）的开发者来说，这种浪费尤其明显。

## 二、Semble 的解法：精准切片 + 双路检索 + 智能重排

Semble 的核心思想很简单：只给 Agent 它真正需要的代码片段。为了实现这一点，它采用了三层架构：

### 1. 代码感知分块
Semble 使用 Chonkie 库将代码文件切分成语义连贯的块，而不是简单的行数分割。这意味着一个函数定义、一个类实现会被保持在一个 chunk 中，而不会被粗暴切断。

### 2. 双路检索
对每个查询，Semble 同时运行两条检索路径：
- **语义检索**：使用静态 Model2Vec 嵌入，基于 code-specialized 的 potion-code-16M 模型。这是轻量级的，不需要在查询时进行 transformer 前向传播。
- **词法检索**：使用 BM25 匹配标识符和 API 名称，适合精确符号搜索。

这两条路径的结果通过 RRF（Reciprocal Rank Fusion）融合。

### 3. 代码感知重排
这是 Semble 的亮点。它使用一系列规则对检索结果进行精细化排序：
- **自适应权重**：符号类查询（`Foo::bar`、`_private`）给予更多词法权重，自然语言查询则保持平衡。
- **定义提升**：定义查询符号的 chunk 排在引用它的 chunk 之上。
- **词干匹配**：`parse config` 会同时匹配 `parseConfig`、`ConfigParser`、`config_parser`。
- **文件连贯性**：来自同一文件的多个匹配结果会被整体提升，避免单个孤立的 chunk 被排在前面。
- **噪音惩罚**：测试文件、compat/legacy 适配层、示例代码、.d.ts 声明存根会被降权。

## 三、性能表现：99% 的质量，200 倍的速度

根据官方基准测试，在 63 个代码库、19 种语言、约 1250 个查询上的测试结果显示：

- **NDCG@10**：0.854（CodeRankEmbed Hybrid 为 0.862，仅差 1.4%）
- **索引时间**：263ms（CodeRankEmbed Hybrid 为 57s，快 218 倍）
- **查询时间**：1.5ms（CodeRankEmbed Hybrid 为 16ms，快 11 倍）

值得注意的是，CodeRankEmbed Hybrid 是一个 137M 参数的模型，而 Semble 的核心是静态嵌入 + 传统检索算法。这意味着 Semble 在查询时不需要 GPU、不需要 API key、不需要外部服务，完全本地化运行。

## 四、Token 效率：98% 的节省

这是 Semble 最吸引人的卖点。官方数据显示：

- Semble 平均节省 **98% 的 token**
- 在 2k token 预算下达到 94% 召回率
- grep+read 需要 100k 上下文窗口才能达到 85% 召回率

Semble 甚至提供了一个 `semble savings` 命令，可以统计你节省的 token：
```
Period        Calls   Savings
────────────────────────────────────────────────────────────────
Today         42      ~58.4k tokens (95%)
Last 7 days   287     ~312.4k tokens (90%)
All time      1.4k    ~1.2M tokens (89%)
```

对于高频使用者，这些节省会累积成可观的成本降低。

## 五、集成方式：MCP 服务器 + Bash 工具

Semble 提供了两种集成方式，覆盖了不同场景：

### MCP 服务器
支持 Claude Code、Cursor、Codex、OpenCode 等主流 Agent 平台。安装后，Agent 可以直接调用 `search` 和 `find_related` 工具，无需手动配置。

### Bash 工具
对于无法直接调用 MCP 的 sub-agent（如 Claude Code 的 sub-agents），Semble 提供了 Bash 命令行接口。只需在 AGENTS.md 中添加几行配置，Agent 就可以通过 `semble search` 命令查询代码。

这两种方式都支持本地路径和 Git URL，远程代码库会被按需克隆和索引，索引结果会在会话期间缓存。

## 六、思考：Agent 基础设施的演进

Semble 的出现反映了 AI Agent 基础设施的一个重要趋势：从通用工具向专用优化演进。

早期的编程 Agent 工具主要沿用人类开发者的工作流——grep、读取文件、在编辑器中跳转。但这些工具是为人类设计的，人类可以快速浏览大量代码并过滤噪音，而 Agent 每读一个 token 都是成本。

Semble 意识到了这一点：为 Agent 设计的工具，应该针对 Agent 的约束（token 成本、上下文限制、实时性需求）进行优化，而不是简单复用人类工具。

这个趋势在其他领域也在发生：RAG 系统针对 LLM 的上下文窗口优化、向量数据库针对检索速度和精度优化、代码生成工具针对 Agent 的调用习惯优化。

Semble 的另一个值得注意的设计是：它不追求"完美"，而是追求"够用且快"。NDCG@10 只比顶级模型差 1.4%，但速度快 200 倍。这种实用主义在 Agent 基础设施设计中会越来越重要——对大多数应用场景，"足够好的实时结果"比"完美的离线结果"更有价值。

## 七、适用场景与局限

**适用场景：**
- 中大型代码库的日常探索
- 需要频繁查询特定功能的开发流程
- 对 token 成本敏感的项目
- 需要本地化、隐私保护的场景

**局限：**
- 对于需要精确字符串匹配的场景，ripgrep 等传统工具仍然更可靠
- 对于非常新的框架/库，语义检索可能不如有经验的开发者直观
- 需要额外的学习成本（虽然安装很简单，但团队需要适应新的工作流）

## 八、结语

Semble 的成功（HN 123 分，37 条评论）说明了一个事实：开发者开始认真思考"如何为 AI Agent 设计工具"，而不仅仅是"如何让 AI Agent 适应现有工具"。

这不仅是成本问题，更是效率问题。当 Agent 可以在一毫秒内找到它需要的代码片段，而不是读取几KB无关内容后才开始理解，整个开发流程都会加速。

Semble 的设计哲学值得借鉴：理解 Agent 的真实约束（token、上下文、实时性），然后针对这些约束进行专门优化，而不是让 Agent 在为人类设计的工具中挣扎。
