# 多代理开发是一个分布式系统问题：AGI 也救不了你

> 源自：Kiran Gopinathan 的博客《Multi-agentic Software Development is a Distributed Systems Problem》
> HN 得分：102 | 评论数：54
> 时间：2026-04-14

---

## 问题陈述

当人们谈论多代理 LLM 系统时，一种常见的态度是："等几个月就好了，下一代模型会更聪明。"

Kiran Gopinathan 作为一名形式化验证研究者，对此持强烈反对态度。他的核心论点是：**多代理软件开发本质上是一个分布式系统问题，这个问题不会因为模型变得更聪明而消失**。

这不是当前模型能力的临时缺陷，而是这个领域的固有属性。

---

## 形式化模型：为什么多代理开发需要共识？

假设你对 Claude 说："给我做一个食谱追踪应用。不要出错。"

用数学语言描述，给定提示词 P，我们可以定义 Φ(P) 为所有与提示词一致的软件集合：

```
Φ(P) := { φ | φ 是程序 ∧ φ 与提示词 P 一致 }
```

关键点在于：**自然语言提示词本质上是欠指定的**。同一个提示词可以对应多个有效的程序实现。

当你使用单个 LLM 构建软件时，你是在让它从这个集合中选择一个元素。

但当你使用多个代理 A₁...Aₙ 协作时，问题就变了。你要求每个代理产生软件组件 φ₁...φₙ，使得这些组件都"精化"同一个一致的提示词解释：

```
C(φ₁, ..., φₙ) := ∃φ ∈ Φ(P), ∀i, φᵢ 精化 φ
```

**这就是一个分布式共识问题。**

---

## 为什么是并发问题？

Gopinathan 预判了三个可能的反驳：

1. **"为什么 P 是欠指定的？"**
   - 自然语言本身就有歧义性。如果你想给出精确、无歧义的软件规范，那就用编程语言。除此之外，任何描述都给代理留下了设计决策的空间。

2. **"这个问题真的是并发的吗？"**
   - 一旦你决定让多个代理协作，问题本质上就是并发的——无论它们是并行运行还是在单线程上交错执行。你必须解决协调问题。

3. **"不能用一个 supervisor 来做决策吗？"**
   - 比如：代理并行提交提案，然后有一个 supervisor 管理 PR 合并？
   - Git 仓库确实是协调并行开发的标准方法，但你没有解决根本的并发问题，只是硬编码了一种特定的并发选择，而且不是很好的选择。
   - 当一个设计决策被选中并合并到代码库后，需要 rebase 的工作会怎样？如果有冲突怎么办？有些工作必然丢失。

Gopinathan 的观点不是说多代理开发不能做——人们确实在用代理发布真实的软件。他的观点是：**人们目前是通过临时机制隐式地解决这些协调问题，而这些机制没有经过深思熟虑的保证或失败模式。**

## FLP 定理：安全性、活性、容错性，选两个

Fischer, Lynch, Paterson 的经典论文《一个故障进程的分布式共识不可能性》确立了分布式系统中的基本不可能结果。

**FLP 定理（1985）：** 在任何具有任意消息延迟且至少有一个节点可能崩溃的分布式系统中，没有确定性协议能保证所有非故障节点在有限时间内达成共识。

这个定理的假设对多代理系统适用吗？Gopinathan 认为适用：

1. **异步消息**：LLM 之间的消息传递通常是异步的。LLM 决定何时读取消息，并且可以无限期推迟。无论通过什么机制传递消息，对方何时看到这个消息通常由对方 LLM 决定——当它完成推理轨迹时，或它调用的程序/工具完成时。我们无法对消息交付设置界限。

2. **崩溃故障**：代理非常容易"自己给自己挖坑"。在这个上下文中，崩溃故障不只是进程死亡，还包括代理运行一个永不终止的工具（例如一个有意外循环的 bash 脚本），或者代理直接 `pkill` 自己。代理可能会在没有警告的情况下崩溃。

**这意味着什么？**

在任何多代理系统中，无论代理有多聪明，它们永远无法同时保证：
- **安全（Safety）**：产生符合用户规范的格式良好的软件
- **活性（Liveness）**：总是能就最终软件模块达成共识

你可能对活性部分感到惊讶，因为代理在大多数工作流中似乎都在取得进展并终止。但这不意味着它们达成了共识——实际上，我看到的一个非常常见的模式是**循环往复**：一个代理选择一个设计决策，另一个代理撤销更改，选择另一个决策，然后它们继续循环。

Gopinathan 还提出了一个有趣的洞察：当代理在同一台机器上运行时，我们可能比纯崩溃故障模型做得更好。运行像 `ps | grep claude` 这样的命令可能对应于某种**故障检测器**，从中我们可以获得关于共识的更强界限。

Chandra 和 Toueg 的论文《不可靠故障检测器用于可靠分布式系统》在 FLP 基础上证明了：**在 FLP 设置下，当且仅当允许代理访问故障检测器（可能不可靠）时，共识才是可能的。** 如果代理对应进程，且进程存活等同于进度（这不总是成立），那么 `ps` 可能给我们类似的东西。一个可行的启示是：给 LLM 提供检查其他代理活性的工具。

## 拜占庭将军问题：误解的代理

Lamport 的《拜占庭将军问题》提供了第二个不可能结果。

在分布式系统中，**拜占庭节点**是参与者最坏情况故障的建模概念：FLP 等定理假设服务器失败时崩溃并停止参与协议，但拜占庭节点可能这样做，也可能做**任何事情**——它们可能以任意方式偏离协议，比如伪造其他用户的消息，向不应该接收消息的参与者发送消息，或重复先前的消息。

拜占庭故障模式真的对多代理软件开发相关吗？这个故障模型有些保守——我们的代理不会主动试图破坏系统。但 Gopinathan 认为这个模型在我们的设置中仍然有用，因为拜占庭故障似乎是捕获"对输入提示词的误解"的好方式。

一个误解提示词的代理实际上可以被视为协议中的拜占庭代理，主动反对其他参与者试图产生正确的软件系统。在这样的代理存在的情况下，我们还能达成共识吗？

**拜占庭将军定理（Lamport-Shostak-Pease, 1982）：** 在任何具有 f 个可能任意偏离协议的拜占庭代理的同步消息传递系统中，只有在总进程数 n > 3f + 1 时，诚实进程之间才能达成一致。

具体来说，如果我们启动 n 个代理来实现软件系统，如果超过 (n-1)/3 的代理误解提示词，共识就不可能实现。

Lamport 在论文中提出了这个设置下的算法，高层思想是收集所有参与者的投票并取多数。在我们的设置中，集成投票更多是一个理论构造，类比开始有些断裂——软件实现空间足够大，投票不太可能取得多大进展。但我们仍然可以从这个不可能结果中获得一些有用的见解。

**关键观察**：Lamport 的结果是我们可以容忍的最大误解数量的硬界限。这不是更聪明的代理或更好的 LLM 能解决的问题。我们的综合问题本质上是欠指定的，代理总可能有误解。从这个分析中可操作的启示是：虽然我们不能提高可容忍的误解数量 f，但我们可以通过采用**减少误解数量**的技术来增加成功可能性——特别是外部验证机制，如测试、静态分析和验证，有效地将误解转化为崩溃故障——代理不再基于对自然语言描述的误解继续执行，而是可以崩溃或优化其解释以匹配测试，从中我们可以复用更弱的 FLP 结果。

## 进一步思考

除了 FLP 和拜占庭将军，还有几个其他关键的分布式系统结果值得在多代理软件开发背景下思考：

- **公共知识（Halpern & Moses, 1990）**：定义了在分布式系统中推理认知知识流动的形式化模型，并建立了此类系统中存在公共知识的不可能结果。

- **部分同步（Dwork, Lynch & Stockmeyer, 1988）**：Dwork 等人的部分同步模型是标准逃生舱，尽管 FLP 仍提供共识。关键技巧是对消息延迟设置上界——在真实分布式系统中，设备不会无限期等待消息到达，会超时并请求重发。有了这个简化，共识在增加消息复杂度的代价下变得可解。

- **CAP 定理（Gilbert & Lynch, 2002）**：任何关于分布式系统的"流行"文章都不能不引用著名的 CAP 定理！CAP 像 FLP 一样提供了不可能结果，指出任何分布式系统最多只能选择一致性、可用性和分区容忍性中的两个。

## 结论

那么，这把我们带到了哪里？

Gopinathan 在整篇文章中构建的论点是：**多代理软件开发本质上是一个分布式系统问题，因此它继承了文献中的不可能结果。** FLP 定理告诉我们分布式系统不能同时具有安全性、活性和容错性。Lamport 的拜占庭将军告诉我们共识不可能之前的拜占庭进程的精确界限。注意，这些结果都不依赖于代理的智能。

重点不是说多代理软件开发是不可能的——人们确实在用代理发布真实软件，而且明天可能发布更多。Gopinathan 的论点更多是：这些人目前是通过**隐式地解决这些协调问题**，通常通过临时机制，没有经过深思熟虑的保证或失败模式。分布式系统文献对所有这些机制有精确的形式化，四十年的定理告诉你它们能解决什么、不能解决什么，以及何时何地应该应用不同技术。

这把我们带回到引言中的引语："代理协调的最佳解决方案是等几个月。"

也许吧。我可以承认这一点。但即使如此，上述不可能结果不是当前模型能力的临时产物；它们不会随着代理变得更聪明而消失；它们是领域的固有属性。更聪明的代理可能会缩小我们算法中的常数，但它们不会、也不能消除界限。

如果我们想让多代理软件开发真正扩展，那么迟早有人必须做实际的工作：设计将底层协调问题作为一等公民解决的协议、语言和工具，而不是希望它自己消失。

---

## 实践启示

1. **不要等待 AGI 解决协调问题**：这是分布式系统的根本问题，与代理智能程度无关。

2. **设计显式的协调协议**：而不是依赖隐式的临时机制。

3. **引入外部验证**：测试、静态分析、验证可以将"误解"转化为"崩溃"，使问题更容易处理。

4. **考虑故障检测器**：给代理提供检查其他代理活性的工具。

5. **接受不可能结果**：你不能同时拥有安全性、活性和容错性。做有意识的权衡。

## 文章价值

这篇文章的价值在于它提供了一个**理论框架**来理解多代理系统的根本限制。它不是在说"多代理系统不能工作"，而是在说"如果你想让它可靠地工作，你需要认真对待分布式系统问题"。

对于正在构建多代理系统的人来说，这是一个重要的提醒：不要指望模型进步会自动解决协调问题。这是一个工程问题，需要工程解决方案。
