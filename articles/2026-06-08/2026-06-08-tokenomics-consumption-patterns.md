## 背景

随着基于 LLM 的多智能体系统在软件工程中越来越流行，一个关键问题浮现：它们到底消耗了多少资源？成本和环境影响难以预测，阻碍了实际应用。

这篇论文《Tokenomics: Quantifying Where Tokens Are Used in Agentic Software Engineering》试图回答这个问题。

## 研究方法

作者分析了 ChatDev 框架在 30 个软件开发任务中的执行轨迹，使用的模型是 GPT-5（推理模型）。他们将 ChatDev 的内部阶段映射到标准的软件开发生命周期（SDLC）：

1. Design（设计）
2. Coding（编码）
3. Code Completion（代码补全）
4. Code Review（代码审查）
5. Testing（测试）
6. Documentation（文档）

然后统计了每个阶段的 token 消耗，包括输入、输出和推理 token。

## 核心发现

### 1. Code Review 占大头

最反直觉的发现：**代码审查阶段消耗了 59.4% 的 token**。

这意味着多智能体系统的主要成本不是"写代码"，而是"审查和修正代码"。智能体之间反复迭代，互相挑错，这个过程消耗了大部分资源。

### 2. 输入 token 占主导

输入 token 占平均消耗的 53.9%。这暴露了智能体协作中的低效问题：每个智能体都需要重新读取上下文，导致大量重复输入。

想象一个场景：
- Agent A 生成代码
- Agent B 审查时，需要重新读取整个代码 + 需求
- Agent C 修改时，又需要读取前两轮的上下文
- Agent D 审查时，再次读取...

这种"每次都从头开始"的模式导致输入 token 激增。

### 3. 成本不在首次生成

研究结果表明：智能体软件工程的主要成本**不在初始代码生成，而在自动化的精炼和验证**。

这与直觉相反——很多人以为 LLM 写代码是最贵的部分，但真正烧钱的是让它自己审查、修改、再审查。

## 问题的本质

### 智能体协作的低效

多智能体系统的设计初衷是"分工合作"：
- 一个负责设计
- 一个负责编码
- 一个负责测试
- 一个负责审查

但这种分工带来的问题是：**信息传递成本高**。

传统软件开发中，人类开发者会共享上下文——你改了代码，我只需要看 diff。但 LLM 智能体没有这种共享机制，每次都需要完整输入上下文。

### 缺少增量更新能力

如果智能体能只读取"增量"（比如只看本次审查中发现的问题），输入 token 会大幅降低。但目前的系统设计没有这种能力。

### 过度审查

59.4% 的 token 消耗在代码审查上，说明智能体在反复审查同一个代码，可能是因为：
- 每次审查的标准不一致
- 缺少"已确认无误"的机制
- 审查智能体过于谨慎

## 启示与建议

### 1. 优化智能体协作协议

未来的研究应该开发更 token 高效的智能体协作协议：
- 增量上下文传递
- 上下文缓存
- 差异化审查（只审查修改部分）

### 2. 减少审查轮次

建立更智能的审查机制：
- 如果连续 N 次审查未发现问题，自动停止
- 审查智能体之间共享"已确认的问题列表"
- 按问题重要性排序，优先解决关键问题

### 3. 成本预测

论文提出的方法可以帮助从业者预测成本。如果你知道：
- 项目规模
- 智能体数量
- 预期审查轮次

就可以估算 token 消耗和成本。

## 批判与局限

### 样本规模小

只分析了 30 个任务，样本偏小。不同类型的任务（Web 开发 vs 系统编程 vs 嵌入式开发）可能有不同的 token 消耗模式。

### 只研究了一个框架

ChatDev 是一个特定的多智能体框架。其他框架（如 AutoGPT、BabyAGI）可能有不同的协作模式和 token 消耗特征。

### 未考虑模型选择

研究只用了 GPT-5（推理模型）。不同模型（GPT-4、Claude、开源模型）的 token 消耗模式可能不同。

### 缺少优化实验

论文只是"测量"了 token 消耗，没有尝试"优化"它。比如：
- 增量上下文传递能节省多少 token？
- 减少审查轮次对代码质量有多大影响？
- 智能体数量与 token 消耗的关系是什么？

## 实践意义

如果你正在开发或使用多智能体软件工程系统：

1. **监控 token 消耗**：使用类似论文中的方法，追踪每个阶段的 token 消耗
2. **审查轮次**：设置最大审查轮次，避免无限循环
3. **上下文缓存**：尽可能复用已处理的上下文
4. **成本预算**：根据任务规模预估成本，避免超支

## 总结

这篇论文最重要的贡献不是"发现了代码审查占 59.4% 的 token"，而是**建立了一个可复用的评估框架**。

之前没人系统性地测量过智能体软件工程的 token 消耗，这篇论文填补了这个空白。它告诉我们：
- 成本不在首次生成，而在反复精炼
- 输入 token 占主导，暴露协作低效
- 需要更智能的协作协议

这个方向的研究才刚刚开始。未来可能会出现：
- 更 token 高效的智能体框架
- 成本优化的最佳实践
- 标准化的成本预测工具

如果你关心 AI 软件工程的可持续性（无论是成本还是环境影响），这篇论文值得一读。
